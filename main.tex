\begin{abstract}

The dynamics of a binary black hole systems are governed by their masses and spins. For a binary system it is not possible to define these quantities in an unambiguous way. But, we have a lot of reasonable definitions that reduce to the expected values in the limit of infinite separation. For the simulations done in SpEC one of the spin definition that we use is AKV(Approximate Killing Vector) spins, which as the name suggests are built using the approximate killing vectors of the apparent horizons. Finding these approximate killing vectors requires solving a generalized eigenvalue solve, which scales as O($L^6$), where L is the highest spherical harmonic mode used to represent the apparent horizon. This scaling means that the cost of computing the AKV spins gets very high as we try to simulate more complex systems at higher resolutions. In this paper we describe a new algorithm that uses iterative eigenvalue solvers and runs much faster than the current algorithm using dense solver.


\end{abstract}

\section{\label{sec:Introduction} Introduction}

With the new generation of gravitational wave detectors coming up \cite{Evans:2023euw,2017arXiv170200786A} we will need to redo a lot of our simulations at much higher resolutions. Often doing a simulation at higher resolutions is just a matter of using more computing resources, but often a lot of other issue show up that were not present before. In our case one of the issue that started showing up when trying to simulate more complicated systems was the increasing wall time cost of the spin computations. Unlike the rest of the simulation the spin computations are done on a single core, this means that when it is slow all the other cores are effectively wasting resources waiting for it to complete.

In this paper Section \ref{sec:AKV spin definition} has some mathematical background about how the AKV spins are defined. Then, in Section \ref{sec:Current AKV algorithm} we briefly describe the current algorithm and why it slows down rapidly as the systems being evolved get complicated. In Section \ref{sec:Sparse AKV algorithm} we describe the new algorithm based on ARPACK, which is much faster. Finally, in Sections \ref{sec:Future improvements} and \ref{sec:Conclusions} we talk about possible future improvements and conclude.

\section{\label{sec:AKV spin definition} AKV spin definition}
In this section we only define the quantities relevant to understand the algorithm. Reader should refer to Appendix A of \cite{lovelace_binary-black-hole_2008} and \cite{Owen:2017yaj} for more background and references. There is no unambiguous way to define spin of a dynamical black hole horizon in a bbh system. We usually settle for quasilocal spin definition which reduce to the ``expected" values in the limit of infinite separation. AKV spin ($\chi_{AKV}$) is one of these quasilocal spin definitions which is constructed by using approximate killing vectors of the black hole horizons.

We start by setting up the conventions used throughout the paper. We work in the standard 3+1 decomposition \cite{Baumgarte:2010ndz}, with spatial slice denoted by $\Sigma$. The spatial metric, extrinsic curvature and the two dimensional apparent horizon are denoted by $g_{ij}$, $K_{ij}$ and $\mathcal{H}$. Lower case Latin letter index the spatial tensors and upper case Latin latter index the tensors on the two dimensional horizon surface. We also put a ring on top of the geometric quantities related to $\mathcal{H}$ to separate then from the geometric quantities on the spatial slice $\Sigma$.

The standard definition (CITE) of spin angular momentum ($S$) in given by, 
\begin{align}
S=\frac{1}{8 \pi} \oint_{\mathcal{H}} \phi^i s^j K_{i j} d A
\label{eqn:spin angular momentum}
\end{align}

where  $s^j$ is the outgoing normal of $\mathcal{H}$. And $\vec{\phi}$ is an ``azimuthal" vector field, tangent to $\mathcal{H}$ which carries the information about the spin axis.

It can be shown that \ref{eqn:spin angular momentum} is conserved (CITE) when $\vec{\phi}$ is a Killing vector of the dynamical horizon world tube. For an isolated Kerr black hole this Killing vector defines the spin axis of the horizon, but there is no reason why such a Killing vector field should exist for the horizon of a black hole in a binary system. The next best thing one can use is the vector field which is closest to being a Killing vector field, in other words an approximate Killing vector. We will now describe the method used in SpEC for constructing these approximate Killing vector fields (CITE). We start from the Killing's equation for a vector field $\phi_{B}$
\begin{align}
D_{(A} \phi_{B)}=0
\end{align}

This condition can be spilt into two conditions: first that the field has no expansion
\begin{align}
\Theta:= \mathring{g}^{A B} D_A \phi_B=0
\end{align}

and second that it has no shear
\begin{align}
\sigma_{A B}:=D_{(A} \phi_{B)}-\frac{1}{2} \mathring{g}_{A B} \Theta=0
\label{eqn:shear in terms of phi}
\end{align}

where $\mathring{g}_{AB}$ is the metric on the apparent horizon and $D_A$ is the torsion free covariant derivative associated with that metric. If $\vec{\phi}$ is not a Killing vector field then we can't satisfy both of these condition at once, so we will choose a $\vec{\phi}$ which is expansion free and has the smallest possible shear. Any expansion free vector field tangent to the apparent horizon can be written as
\begin{align}
    \phi^{A} = \epsilon^{AB}  D_{B} z,
    \label{eqn:phi in terms of z}
\end{align}

where $z$ is some smooth function and $\epsilon^{AB}$ is the Levi-Civita tensor. Now the problem of finding the approximate Killing vector boils down to finding the $z$ which gives the smallest shear or minimizing the norm $\|\sigma\|^2$:
\begin{align}
\|\sigma\|^2:=\oint_{\mathcal{H}} \sigma_{B C} \sigma^{B C} d A .
\end{align}

Using the equations \ref{eqn:shear in terms of phi} and \ref{eqn:phi in terms of z} then integrating by parts we get
\begin{align}
\|\sigma\|^2=\oint_{\mathcal{H}} z H z d A.
\label{eqn:shear norm in terms of H and z}
\end{align}

$H$ is
\begin{align}
H z=D^4 z+\mathring{R} D^2 z+D^4 \mathring{R} D_A z
\label{eqn:equations for H in terms of laplacians}
\end{align}

where $D^2$ is the laplacian on the apparent horizon and $\mathring{R}$ is the Ricci Scalar. If we try to minimize equation \ref{eqn:shear norm in terms of H and z} with respect to $z$ then we will end up with the trivial solution that $z$ is constant and $\vec{\phi}$ is zero. To avoid this we will require the minimized $\vec{\phi}$ has a positive norm,
\begin{align}
\oint_{\mathcal{H}} \phi^A \phi_A d A
\label{eqn:Norm condition to be enforced using Lagrange multiplier}
\end{align}

By using Lagrange multiplier and the fact that $\phi^A \phi_A = D^A z D_A z$ we get
\begin{align}
I[z]:=\oint_{\mathcal{H}} z H z d A+\lambda\left(\oint_{\mathcal{H}} D^A z D_A z d A-N\right),
\end{align}
where $N$ is still an undetermined positive constant. Fixing the value of $N$ is same as normalizing the value of the approximate Killing vector field and getting not just the direction but the magnitude of the spin. We will not be dealing with $N$ in this paper and readers can refer to \cite{lovelace_binary-black-hole_2008} for more info.

Minimizing this functional with respect to $z$ gives a generalized eigenvalue problem:
\begin{align}
  H z = \lambda D^2 z
  \label{eqn:generalized eigenvalue problem}
\end{align}

Now we can try to discretize the equation \ref{eqn:generalized eigenvalue problem} and solve it numerically. The only remaning issue is that both $H$ and $D^2$ still share a kernel where $z$ is a constant, which makes this eigenvalue problem singular. The standard way to deal with such issues is to remove the common kernel space and then solve for z. In our case this is done trivially because we are discretizing using spherical harmonics and can just remove the $Y_{00}$ mode.

Once we have discretized the equation \ref{eqn:generalized eigenvalue problem} the next step involves finding the three smallest eigenvalues and the corresponding eigenvectors. To get the final spin axis and spin magnitude still involves more steps but they are not relevant for this paper and can be found in \cite{Owen:2017yaj} and \cite{lovelace_binary-black-hole_2008}.

\section{\label{sec:Current AKV algorithm} Current AKV algorithm}

Let's call the matrices formed by discretizing the operators $H$ and $D$ in equation \ref{eqn:generalized eigenvalue problem} $M$ and $B$ respectively. Now we have generalized eigenvalue problem $Mx = \lambda Bx$ and getting the eigenvectors is just a matter of using a standard library like LAPACK. To understand why this became an issue we need to understand how big these matrices are and how they grow with increase in the complexity of the apparent horizons. Apparent horizons in SpEC are represented using spherical harmonics. When the black holes are far away from each other their apparent horizons are not too deformed and can be approximated using a small number of spherical harmonics. On the other hand when the black holes are about to merge their horizons become very distorted, and we need a larger number of spherical harmonics to approximate the horizon. We will denote the highest spherical harmonic used in representing the apparent horizon as $L$. $L$ can be as small as 15 for simple non-spinning systems during the inspiral but for some complex cases it can get as large as 90 near the merger.

The number of rows in $M$ and $B$ matrices is $N = (L-1)^2 -1$. We have a $-1$ because we ignore the $Y_{00}$ as describe in the Sec.(\ref{sec:AKV spin definition}), the $L-1$ comes from the fact that we will need to take second order derivatives and that pollutes the two highest modes. We only need the eigenvectors corresponding to the three smallest eigenvalues but using LAPACK's dense solvers (dggev) means we always compute all eigenvalues\footnote{There are variations of Lapack's dggev that can find just the eigenvectors corresponding to the smallest three eigenvalues but they provide very little speedup as explained at https://www.netlib.org/lapack/lug/node70.html. In short, before finding the eigenvectors Lapack does a reduction of the eigensystem and this ends up being the most costly operation. Once the reduction is done finding the eigenvectors itself is comparatively cheap which means that finding just a few eigenvectors is not significantly faster than finding all of them.}. The time complexity of Lapack's dggev solver $O(N^3)$ or equivalently $O(L^6)$, this is the reason why the cost of dggev is irrelevant for most the inspiral until we get close to merger where it starts to increase rapidly, Table \ref{table:timing_data}. This scaling is particularly bad for SpEC where because of the its parallelization infrastructure we have to run dggev on a single core. There were simulations where a single core doing the spin calculations was taking over 30minutes while the rest of the cores were wasting resources waiting for it to finish. This waste of compute resources will only get worse as we need to simulate more complex parts of the parameter space at much higher accuracy for the next generation of detectors. 

\section{\label{sec:Sparse AKV algorithm} Sparse AKV algorithm}

A large part of why ddgev is so slow is because it has to find all the $L^3$ eigenvalues when we just need three. The efficient way to solve these kind of problems is to use iterative solvers like ARPACK.

In SpEC we have two functions let's call them $f_M(x)$ and $f_B(x)$ that give the action of the matrices $M$ and $B$ respectively on a vector $x$. Thus, to generate the matrices we need to operate on $N$ cartesian basis vectors. This means that the time complexity of generating the matrices $M$ and $B$ is $O(N^2)$, one $N$ because we need to act on $N$ basis vectors and another $N$ because the cost of functions $f_M(x)$ and $f_B(x)$ is $O(N)$. In theory, we don't even need to construct these matrices when using ARPACK because it can directly work with the functions $f_M(x)$ and $f_B(x)$ saving both memory and compute. This is shown in Fig.(\ref{fig: ARPACK call graph for nonsymmetric generalized case}), where we just need the functions $f_M(x)$ and $f_B(x)$ to find the eigenvectors. Note that we need to solve a linear problem($y = B^{-1} v$) to get the vector $y$ required by ARPACK, this can be done using iterative solvers like GMRES or BiCGSTAB.

As it turn out we do not want to use ARPACK in the way describe above because it will get very slow and might even fail when $L$ gets large. This is because we want to find the three smallest eigenvalues while the algorithm in ARPACK converges much faster when we solve for the largest eigenvalues. Another peculiarity of the algorithm is that it also converges faster when the eigenvalues are well separated. This can be seen in the next section where we construct an artificial matrix and use ARPACK to find the largest and the smallest eigenvalues.

\subsection{\label{ssec:Speed difference between finding the smallest and the largest eigenvalues} Speed difference between finding the smallest and the largest eigenvalues}

We generate a diagonal matrix with entries $1,4,9,16, \ldots, N^2$.
% Note that the small eigenvalues are clumped together while the large eigenvalue are well separated. 

\begin{lstlisting}[language=Python, caption=Matrix generation code]
import scipy.sparse.linalg as ss
import numpy as np
N = 500
vals = np.arange(1, N + 1, dtype=np.float64)
mat = np.diag(vals**2)
\end{lstlisting}

Now, we ask Arpack to find the largest three and the smallest three eigenvalues of this matrix,

\begin{lstlisting}[language=Python, caption=ARPACK timing comparison]
# LM: largest three
ss.eigs(mat, k=3, which="LM")
#19.6 ms

# SM: smallest three
ss.eigs(mat, k=3, which="SM")
#1.93 s

# Shift-invert near sigma = 0
ss.eigs(mat, k=3, sigma=0.0, which="LM")
#5.5 ms
\end{lstlisting}

We can see that finding the three smallest eigenvalue is much slower than finding the three largest eigenvalues (19.3ms vs 1930ms). But we can do something called Shift-invert transform which can transform the smallest eigenvalues into the largest eigenvalues making the ARPACK much more efficient\footnote{This is just an artificial example to illustrate the importance of shift invert transform and the speedups that one can achieve. In our case, for a black hole with spin $(0,0,0.5)$ we get a speedup of around 5 times for $L = 15$ and around 10 times for $L=55$, but this further increases with the matrix size.}.

% Another byproduct of Arpack being an iterative algorithm is that it can only find the largest n and the smallest n eigenvalues of a problem. Thus, even if we want to find just the third largest eigenvalue of a system Arpack will have to first find the two largest eigenvalues. This can become a substantial cost if we are just interested in the eigenvalues closer to the middle of the spectrum and shift invert transform is a method to deal with this issue.

\subsection{\label{ssec:Shift Invert transform} Shift Invert transform}

Shift invert transform is a method of transforming our eigenvalue problem into another where the eigenvalues that we desire become large and well separated from its neighboring eigenvalues.

Original problem:

\begin{equation}
    M x = \lambda B x
\end{equation}

After shift invert transform:

\begin{equation}
    (M - \sigma B)^{-1} B x = \nu x , 
    \label{eqn:shift_inverted_equation}
\end{equation}  
where,
\begin{equation}
    \nu = \frac{1}{\lambda - \sigma}
    \label{eqn:shift_invert_evals_relation}
\end{equation}


First thing to observe is that both the problems have the same eigenvectors and the corresponding eigenvalues are related via the equation~\ref{eqn:shift_invert_evals_relation}. Note that if we choose $\sigma$ such that it is closer to an eigenvalue $\bar{\lambda}$ that any other eigenvalue, then the corresponding $\bar{\nu}$ will become the largest eigenvalue in the shift inverted system. In the Sec.(\ref{ssec:Speed difference between finding the smallest and the largest eigenvalues}) we saw that the if we want to find the eigensystem corresponding to the smallest eigenvalues then using shift invert transform provides a much effecient way. Fig.(\ref{fig:ARPACK_call_graph_shift_invert_generalized}) show the steps involved when we use shift inverse transform in ARPACK. Note that we start by fixing a value of $\sigma$ and then require the action of $f_{B}(x)$ and $f_M(x) - \sigma f_B(x)$ on the vector $x$ given by ARPACK. We will need to use a linear solver to get the action of matrix $(M - \sigma B)^{-1}$, but the matrix $M - \sigma B$ becomes singular if $\sigma$ ends up being equal to one of the eigenvalues. Even if the $\sigma$ is not exactly equal to one the eigenvalues but still very close we get into a situation where the matrix $M -\sigma B$ becomes very ill conditioned and our linear solver fails or worse gives inaccurate results. 

This is not an issue if we finding these in isolation because we can always change the value of $\sigma$ a little and then try again. In a long running simulation where the eigenvalues keep on changing we can not afford to suddenly have a slightly wrong eigenvectors because one of eigenvalues drifted too close to the value of $\sigma$. Fortunately, if we are using the norm described in Eq.(\ref{eqn:Norm condition to be enforced using Lagrange multiplier}) then the matrices $M$ and $B$ are going to be positive semidefinite and negative semidefinite respectively, which implies that all the eigenvalues $\lambda$ will be negative. Note that not all norms lead to eigenvalue problems where all the eigenvalues are negative, for example the norm used by Cook Whiting in (CITE) will not have this property making it hard to choose a stable and effective value of $\sigma$.

Ideally, we want $\sigma$ to be as close to the smallest eigenvalue as possible. Given all the eigenvalues in our case are guaranteed to be negative we can just set $\sigma$ equal to zero which will make the smallest eigenvalues as large and well separated as possible. Having large and well separated eigenvalues makes the inner Arnoldi iterations in ARPACK very effecient allowing us to find those eigenvalues much faster. Although, it is important to note that as $\sigma$ gets closer to one the eigenvalues the matrix $M - \sigma B$ becomes more and more ill-conditioned. Given we need the action of $(M - \sigma B)^{-1}$ for ARPACK we do not $M-\sigma B$ to become too ill-conditioned. This means that there is an ideal value of $\sigma$ which makes the smallest eigenvalues as big as possible without making the matrix $M - \sigma B$ too ill-conditioned. Based on some tests that we did we decided to go with $\sigma = 0.1$, which has worked well for the systems we have simulated till now.

\subsection{\label{ssec:action-of-shift-invert} Action of \texorpdfstring{$(M-\sigma B)^{-1}$}{(M-sigma B)^(-1)}}

Once we have chosen a reasonable value for sigma the next big thing is how to get the action of matrix $(M-\sigma B)^{-1}$. The first thing that we tried was using iterative algorithms(GMRES and BiCGSTAB) to get the action of inverse. This was particularly attractive in our case because we do not start with matrices but instead construct them by repeatedly calling the functions $f_{M}(x)$ and $f_{B}(x)$. In almost all cases these iterative algorithms require a good preconditioner to converge at an acceptable rate. We tried a few preconditioners and the whole thing worked really well in the early parts but the convergence became very slow in many cases when the horizon was too deformed. Furthermore, constructing non trivial preconditioners from $f_{M}(x)$ is almost as costly as constructing the full matrix itself.

As already mentioned in the section about selecting the value of $\sigma$, given the nature of our problem we would rather have a stable but slightly inefficient algorithm over a mostly stable but fast algorithm. Thus, we decided to eat the cost of constructing the full matrices $M$ and $B$ and then used LU factorization to efficiently apply the action of $(M- \sigma B)^{-1}$.

\section{\label{sec:Future improvements} Future improvements}

The simplest first possibility that may work is going back to the matrix free version and generating the full matrices every 100th step or so to use as a preconditioner. If the spin does not change very rapidly then the matrix should also change slowly and the same preconditioner could be used for multiple time steps. The main issue is that close to the merger spins will change rapidly, and there we may end up calculate the full matrices at every step defeating the whole purpose. All the following suggestion are simply noting the parts that can be parallelized.


Now that we are using Arpack the bottleneck is matrix generation. The matrix is simply generated by using the action of the matrices on unit vectors. In theory the matrix action function can act on multiple basis vectors in parallel.

After, that the next most costly process is the LU decomposition and using it solve a linear system which is being done by Lapack.

Finally, there is Arpack which takes hardly a few seconds to run once the LU decomposition is done. The chances that we would ever need something faster than it is slim but there is a parallel version of Arpack in the \href{https://github.com/opencollab/arpack-ng/tree/master/PARPACK}{Arpack-ng github repo}.
